# 故事审查标准

> **20** 管怎么写。**21** 管怎么审。**postcheck** 管机器底线。
>
> 本项目是一个**可承载多种故事的世界**，不是单一志异短篇流水线。
> 审查只守「像这个世界里的人在过日子、在行走、在动心、在出事」；
> **不**用某一种结尾、某一种怪谈解释、某一种单元剧公式去卡死所有作品。

---

## 〇、世界与故事形态（审查前先认清）

### 世界底色（方向）

- **轻玄幻的武侠底盘**：有江湖事、门派恩怨可写，有刀剑身法与超常意象；**不走**升级打怪数值流。
- **小架空**：以明中后期气质与山川制度为骨架，允许虚构地名与项目原创规矩；人物可把传说当真，作者仍不把神妖异术写成可验证事实。
- **江湖优先**：镖路、仇隙、规矩、饭碗、面子、门派与散人——故事可以很江湖，也可以只沾一点江湖边。
- **可适当糅合现代概念**：组织、契约、舆论、身份焦虑、信息差等**当代可感的结构**可以化进人物处境；禁止的是**裸奔网络梗、短视频腔、职场 PPT 黑话**直接穿古装。
- **形态开放**：日常、情爱、诡异、大事件、多视角诡计、强意象（如悬空之绳摘月一类「纸月亮」式画面）、群像、单人私语……都可以。

### 每篇故事先填一张「形态卡」（写进 01_设定 或 07 报告头）

| 项 | 例 |
|---|---|
| 主类型 | 江湖 / 日常 / 情爱 / 诡异 / 大事件 / 诡计 / 意象志异 / 混合 |
| 视角 | 单一人称 / 多视角 / 限制知 / 不可靠叙述 … |
| 异能与超常 | 无 / 氛围暗示 / 人物坚信或传闻“坐实”（作者仍不确认） |
| 历史锚定 | 严格贴某年某地 / 小架空虚构县 / 年号模糊 … |
| 现代概念融合 | 无 / 有（写明化用了什么结构，而非什么梗） |
| 本篇**不**适用的模块 | 例如：本篇无怪谈 → 跳过 M-怪谈 |

审查时：**只跑「核心底线」+ 形态卡勾选的模块**。未勾选的模块不评分、不否决。

## 一、何时必须审查

| 触发点 | 最低深度 |
|---|---|
| 新故事初稿完成 | 核心 R0–R3 + 形态卡上的模块 |
| 大段重写 / 换类型 / 换视角 | 同上，完整重做 |
| 只改语词、标点、小段对白 | R0；对白动则加抽查声纹 |
| 试读 / 外发前 | 核心 + 已选模块；写或追加 `07_审查报告.md` |

**无怪谈的故事**不缴“怪谈税”；但只要出现怪谈或异象，就必须保留至少一种合理解释，作者不宣布最终答案。克制治愈结尾、官方记录反讽仍是可选写法。

## 二、双视角（始终保留，但问法随类型变）

### 读者：只读正文

1. 愿不愿意看下去？
2. 读完脑子里剩什么（人、事、画面、情绪）？
3. 有没有被作者按头总结？
4. 这个故事**作为它声称的类型**是否成立（江湖是否像江湖，情爱是否动人，诡计是否公平）？
5. 是否还想再进这个世界？

### 编辑：正文 + 本篇设定包

1. 与形态卡是否一致？
2. 人物是否像活人，而非设定说明书？
3. 超常/武/江湖若出现，是否服务于人与事，而非展览设定？
4. 现代概念是「化进去了」还是「穿帮了」？
5. 系列时间线、称呼、许可与状态有无硬伤？

## 三、核心底线（每篇必做 · R0–R3）

### R0 · 机器底线

```bash
python3 scripts/postcheck_story.py stories/SXXX_工作标题/正文.md
# 外发前建议
python3 scripts/postcheck_story.py stories/SXXX_工作标题/正文.md --strict
```

过 R0 再谈人工。词表抓的是**裸奔现代器物/网络梗/重复句**，不是禁止「现代可感的戏剧结构」。

### R1 · 可读与完成度

- 有清晰的阅读承诺（这篇在讲什么类型的事）并基本兑现。
- 不是设定散文冒充故事：要有人物选择、冲突或可感变化（变化可以是心境、关系、立场，不必成长弧）。
- 没有大面积未完成括号、大纲残留、前后直接打架的硬伤。

**门槛**：读者说得出「这是一个××故事」；编辑找不出未写完的大洞。

### R2 · 人物与世界气味

- 主要人物有立场、局限、自己的利害（饭碗、面子、情义、规矩）。
- 世界在故事边角仍在转（哪怕只是一碗面、一封镖信、一句门规）。
- **小架空与武侠元素**允许；但人物说话办事要像活在自家规矩里，而不是 2026 年网文角色开会。

**门槛**：遮住「世界观介绍」，人物仍站得住。

### R3 · 语言与声口（底线版）

- 对白大体有身份差（不必每篇都建完整声纹表，但**不能所有人一个嘴**）。
- 禁止：未消化的网络梗、短视频口头禅、现代职场黑话硬套。
- 允许：把当代可感的处境**翻译**成江湖/市井/官场语言。

**门槛**：无明显穿帮腔；主要人物嘴不完全互换。

> 有 `06_角色声纹表.md` 的篇目，可加做遮名抽样；没有也不强制补表——短篇、诗化、强意象作品可以靠叙述声口成立。

## 四、可选模块（按形态卡勾选）

只审勾了的。未勾 = 不适用 = 不算缺陷。

### M-江湖 / 武戏

- 冲突是否落到规矩、恩怨、饭碗、门派/散人位置，而非只有打戏说明书。
- 武与异术是「事上见」，不是等级表展览。
- 江湖笑话与狠劲是否来自人物认真，而非作者耍帅。

### M-怪谈 / 双解（出现怪谈时必启用）

- **必须**：保留至少一种合理解释；超常读法可以存在，但作者不得宣布最终答案。
- **也可选**：明确「此间有妖/有术/有意象为真」（小架空玄幻允许），但须自洽，且仍服务人物。
- 不要把「必须永不解释」当成全项目唯一神法；**解释与不解释都是手段**，手段要配类型。

### M-强意象（「纸月亮」式）

- 核心意象是否一眼难忘、是否贯穿或收束有力。
- 是否沦为无根的奇观堆砌；奇观后人物是否仍在。
- 允许诗性跳跃，但仍须让读者跟得上情绪逻辑。

### M-多视角 / 诡计

- 视角切换是否有标示与收益。
- 若是诡计/解谜：对读者是否基本公平（线索可回看），还是作者最后变魔术。
- 不可靠叙述须有回报，而非纯折磨。

### M-情爱

- 关系推进是否有具体阻碍与 provable 吸引（性格、处境、选择），而非只靠设定配种。
- 身体与伦理尺度是否自觉，不靠现代人设说教硬掰古人。

### M-日常

- 烟火是否真，有没有「为了写日常而展览民俗」。
- 小波动是否够撑起阅读，结尾是否允许平而不淡。

### M-大事件

- 大历史/大战/灾异是否落到具体人的肩上。
- 避免只有全景航拍没有脸。

### M-现代概念融合

- 写明化用的是哪种结构（如契约精神、舆论反噬、身份文书、组织科层）。
- 检查：读起来是「这个世界里本来就会发生的拧巴」，还是「古装城市中产开会」。
- 过模块 = 化用成功；失败则改写，不因此否定整篇类型。

### M-系列钩子（可选）

- 与其他篇人物、地名、江湖谣言是否故意勾连。
- **不要求**每篇都埋伏笔；独立可读永远优先于宇宙维护。

## 五、红灯（一票否决 · 真正少数）

全项目共通，与类型无关：

1. R0 阻断级未修（时代器物裸奔、网络梗硬塞等脚本 BLOCK）。
2. 全文无法辨认人物利害，纯设定或纯金句展览。
3. 歧视、虐待等描写若出现，无叙事必要且以猎奇为目的（项目底线）。
4. 把**未标原创/架空**的内容伪装成不可质疑的史实训导读者（小架空可以，伪装史料诈骗不行）。
5. 把修炼、法力、等级、超能力或可验证妖鬼谱系写成作者确认的世界事实。
6. 许可与仓库安全红线（见 `AGENTS.md`）。

**下列不再是全项目红灯**（曾误伤开放形态，现降为模块内建议）：

- ~~结尾必须不治愈~~
- ~~必须官方记录反讽~~
- ~~必须单元剧 1–5 万字~~

## 六、黄灯（改不改由作者定，报告里记一笔即可）

- 作者腔总结过多。
- 同一笑点/同一设定讲解重复。
- 档案称呼与正文不一致。
- 现代概念化用边缘穿帮。
- 武戏或意象过长压死人。
- 多视角无标记导致误读。

黄灯**不阻止**「基础通过」。

## 七、结论、审印与网页状态

审查结论与网页审印是同一套状态，不再用“已有正文/已录”冒充“已经审过”。

| `review.state` | 审印 | 含义 | 网页表现 | 是否算过关 |
|---|---|---|---|---|
| `passed` | **验讫** | R0–R3 通过；启用模块无必须返修项 | 朱印「验讫」 | **是** |
| `passed_with_notes` | **朱注** | 核心通过，但有明确黄灯或外发前修订项 | 朱印「朱注」 | **是，附注** |
| `revise` | **退修** | 存在 S1/S2、红灯，或核心/主章公约未过 | 破印「退修」 | **否** |
| `pending` | **待勘** | 未完成当前版本人工审查，或审印已失效 | 素印「待勘」 | **否** |
| `exempt` | **卷宗** | 非故事正文，或不适用故事审查 | 灰印「卷宗」 | 不计 |

### 7.1 “过关”的最低条件

只有同时满足下列条件，才能写入 `passed` / `passed_with_notes`：

1. 当前 `正文.md` 的 `Version` 与 `review.source_version` 一致；
2. R0 严格模式通过；
3. 人工 R1–R3 完成；
4. 形态卡启用模块完成；
5. 主线章额外通过 `docs/原则/25_踏线人主章公约.md`；
6. 外发/连载条目额外核对 `docs/原则/24_卷册与连载阅读体验.md`；
7. 报告路径存在，且列出红灯、黄灯和结论。

机器 PASS **不等于**人工过关；`status: 基础通过` 也不自动生成审印。

### 7.2 审印失效规则

出现下列任一情况，条目应先退回 `pending`，复审后再盖印：

- `正文.md` Version 变化；
- 大段重写、换视角、换类型、换结尾；
- 关联设定发生会影响本篇的重大变化；
- catalog 的 `track` / `main_path` 改动导致本章功能变化；
- 上游章节改动造成知识、因果、人物关系穿帮。

只改错别字、标点或不改变含义的小句，可保留审印，但报告需记一次“轻改核验”。

### 7.3 问题严重度

| 级别 | 含义 | 对结论的影响 |
|---|---|---|
| **S1** | 世界规则、历史红线、主因果或人物根本动机崩坏 | 必须 `revise` |
| **S2** | 明显影响章节独立性、人物可信度、连载留存或主副线边界 | 通常 `revise` |
| **S3** | 局部重复、声口趋同、时代词或节奏问题 | 可 `passed_with_notes`，外发前修 |
| **S4** | 风格建议、可选增强 | 不阻止通过 |

### 7.4 审印数据（`stories/catalog.json`）

每个可阅条目使用独立 `review` 对象：

```json
{
  "review": {
    "state": "passed_with_notes",
    "seal": "朱注",
    "round": 1,
    "date": "2026-07-18",
    "scope": "base",
    "standard": "docs/原则/21_故事审查标准.md",
    "standard_version": "0.3",
    "report": "stories/审查总簿.md",
    "reviewer": "Codex·总编审",
    "source_version": "1.0",
    "note": "核心通过；外发前需收束重复说明。"
  }
}
```

- `status` 管稿件阶段；`review.state` 管审查结论，二者不得混用。
- 网页只根据 `review` 盖审印；缺字段一律显示「待勘」。
- 审印必须醒目但不能遮挡标题；推荐使用朱砂方印、破印和素印。

## 八、报告怎么写（轻量）

单篇长稿优先：`stories/<地脉组>/<编号_标题>/07_审查报告.md`
短条目/整卷可合订：`stories/<地脉组>/07_组审查.md` 或 `stories/审查总簿.md`
模板：`stories/_templates/07_审查报告.md`

每轮追加一节即可，建议结构：

```markdown
## 审查 · YYYY-MM-DD

- 形态卡：（类型 / 视角 / 超常 / 架空度 / 现代概念 / 启用模块）
- R0：PASS / FAIL
- 核心 R1–R3：一句话各
- 模块：只写启用的
- 红灯 / 黄灯
- 严重度：S1 / S2 / S3 / S4
- 结论：验讫 / 朱注 / 退修 / 待勘
- review：source_version / round / report
- 修改单：（可选，P0 才强制）
```

评分表**可选**，不爱打分可以只写结论。

## 九、工作流（收束版）

```text
形态卡 → 写（20 + 本篇 01–06 按需）→ 正文
  → R0 postcheck
  → 核心 R1–R3
  → 仅跑形态卡上的 M-模块
  → 07 报告或审查总簿
  → 红灯/S1-S2 退修；黄灯可朱注
  → 写入 catalog.review（版本绑定）
  → 网页显示审印 → 更新 11 状态等
```

与 S001 类「制度志异」适配：可勾 M-怪谈、M-日常。
与江湖刀剑、强意象、多视角诡计、情爱篇：换模块即可，**不必模仿 S001 的结尾哲学**。

| 来源 | 如何用 |

## 十一、快速对照卡

**每篇 20–40 分钟基础审：**

1. 核对正文 Version 与旧审印是否失效
2. 跑 postcheck --strict
3. 填/核形态卡
4. 读者通读：剩什么？像不像这类故事？
5. 编辑：人物利害、穿帮腔、红灯
6. 主线加查重复增量与“懂/债/名”
7. 写报告，给审印并更新 catalog.review

**不要：**

- 用 S001 的制度反讽当唯一正确结尾
- 给没有怪谈的故事强塞双解；有怪谈时则必须保留合理解释
- 把审查写成比正文还长的论文
